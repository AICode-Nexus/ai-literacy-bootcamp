# 设计文档：Claude Code Skills 实战专题教程

## 概述

为 AI 素养训练营新增一节独立专题课程，教非技术用户从零掌握 Claude Code Skills 的使用与编写。

## 定位

- **课程类型**：独立实战专题（不编号，不纳入主课程体系）
- **前置知识**：零前置，课内自包含所有必要背景
- **目标受众**：非技术用户（不会写代码，但日常使用 AI 工具）
- **终点能力**：能从零编写一个完整可用的自定义 Skill
- **预估时长**：约 3 小时
- **教学风格**：讲解 ≤ 40%，实操 ≥ 60%，每个 Section 有复制即用模板
- **Claude Code 版本**：基于 Claude Code CLI 最新稳定版（2026 年），Skill 功能需 Claude Code 已支持自定义 Skills

## Skill 技术规格

Claude Code Skills 是放在特定目录下的 `.md`（Markdown）文件。Claude Code 启动时自动加载这些文件作为行为指令。

**文件位置**：
- 全局 Skills：`~/.claude/skills/` — 所有项目通用
- 项目 Skills：`项目根目录/.claude/skills/` — 仅当前项目生效

**调用方式**：
- 用户通过 `/skill-name` 斜杠命令显式调用（skill 文件名去掉 `.md` 后缀即为命令名）

**完整 Skill 文件示例**：

```markdown
# 会议纪要整理

把会议录音转写的原始文字整理成结构化的会议纪要。

Use when the user asks to "整理会议纪要", "meeting notes", or pastes a block of meeting transcript text.

## 工作流程

1. 识别参会人员（从对话中提取发言者）
2. 提取核心议题（按讨论顺序列出）
3. 归纳每个议题的结论/决议
4. 提取所有待办事项，标注负责人和截止日期
5. 过滤口水话、重复内容、无意义的过渡语

## 输出格式

按以下结构输出：

### 会议信息
- 日期：[从文本推断或标注"未知"]
- 参会人：[逗号分隔]

### 议题与决议
[编号列表，每个议题一段]

### 待办事项
| 事项 | 负责人 | 截止日期 |
|------|--------|----------|
| ...  | ...    | ...      |

## 约束

- 总输出不超过原文的 30%
- 不添加原文中没有的信息
- 如果无法确定负责人或截止日期，标注"待确认"
- 使用中文输出
```

**关键要素说明**：
| 要素 | 作用 | 是否必须 |
|------|------|----------|
| 标题（`# 标题`） | Skill 的名称，也是 `/` 命令名的来源 | 是 |
| 描述段落 | 一句话说明这个 Skill 做什么 | 是 |
| 触发说明（`Use when...`） | 告诉 Claude 什么情况下使用此 Skill | 推荐 |
| 工作流程 | 分步骤的执行指令 | 是 |
| 输出格式 | 定义输出的结构 | 推荐 |
| 约束 | 边界条件和限制 | 推荐 |

**成功标准**：一个"完整可用"的 Skill 意味着——用 `/skill-name` 调用后，输出符合预期格式，且连续 3 次调用结果质量稳定。

## 课程结构

### Section 1：概念引入 — Skill 是什么（15 min）

**目标**：让学员理解 Skill 的本质和价值。

**核心类比**：Skill = 菜谱，Claude Code = 厨师。
- 没有菜谱，厨师也能做菜，但你得每次口头描述想吃什么
- 有了菜谱，一句"做这道菜"就够了，结果稳定、可复用

**内容要点**：
1. 什么是 Claude Code（一句话：终端里的 AI 助手，能读写文件、执行命令）
2. 什么是 Skill（一句话：一个 `.md` 文件，里面写好了指令，让 Claude Code 按特定方式完成特定任务）
3. Skill 解决的痛点：
   - 每次都要重复输入长提示词
   - 输出质量不稳定
   - 复杂工作流记不住步骤
4. Skill vs 普通提示词的区别：

| 维度 | 普通提示词 | Skill |
|------|-----------|-------|
| 存储 | 每次手动输入或从笔记复制 | 保存为文件，永久可用 |
| 调用 | 粘贴一大段文字 | 输入 `/名字` 一个命令 |
| 稳定性 | 每次措辞不同，结果波动 | 指令固定，输出一致 |
| 分享 | 发消息给同事 | 放进项目目录，团队自动共享 |
| 复杂度 | 适合单步任务 | 可编排多步骤工作流 |

**产出**：学员能用自己的话解释"Skill 是什么"，并说出至少 2 个使用场景。

---

### Section 2：体验现成 Skill（20 min）

**目标**：快速体验自定义 Skill 的效果，建立直观感受。

**前置准备说明**：
- 确认已安装 Claude Code（提供安装指引，含常见安装问题排查）
- 确认终端可正常运行
- 课程提供一个预配置的练习目录，内含 2-3 个示例 Skill 文件

**注意**：Claude Code 有内置命令（如 `/help`、`/clear`）和用户自定义 Skill（`.md` 文件）。本课程聚焦的是后者——用户自己编写的 Skill。

**实操步骤**：
1. 打开终端，启动 Claude Code
2. 下载课程提供的练习包（含预写好的示例 Skill）
3. 将示例 Skill 文件放入 `.claude/skills/` 目录
4. 体验 2-3 个预装的自定义 Skill，例如：
   - `/meeting-notes` — 会议纪要整理（粘贴一段示例文本）
   - `/content-rewrite` — 内容改写（输入一段文字 + 目标平台）
   - `/daily-summary` — 日报生成（基于当天的笔记文件）
5. 观察：输入了什么？输出了什么？格式是怎样的？

**教学重点**：
- 调用 Skill 就像点菜：`/skill-name` + 可选参数
- 每个 Skill 都是一个 `.md` 文件——等会儿我们会打开看它的内容
- 让学员感受"一个斜杠命令完成原本需要 5 分钟描述的任务"

**常见问题处理**：
- Claude Code 未安装：提供逐步安装指引（macOS / Windows / Linux）
- 终端权限问题：提供 chmod 命令说明
- Skill 未被识别：检查文件是否在正确目录、文件名是否有空格

**产出**：学员成功调用至少 2 个 Skill 并看到输出。

---

### Section 3：解剖一个 Skill（25 min）

**目标**：理解 Skill 文件的结构，消除"这是代码"的恐惧。

**核心类比**：Skill 文件 = 一份结构化的工作说明书，用的是 Markdown（和写笔记一样的格式）。

**解剖对象**：使用本课程"Skill 技术规格"部分的"会议纪要整理"完整示例。

**逐块讲解**：

```
一个 Skill 文件的组成：

┌─────────────────────────────────┐
│  # 标题                          │  ← 名字，也是 /命令名
├─────────────────────────────────┤
│  一句话描述                      │  ← 这个 Skill 做什么
├─────────────────────────────────┤
│  Use when...                     │  ← 什么时候用（触发说明）
├─────────────────────────────────┤
│  ## 工作流程                     │  ← 分步骤的执行指令
├─────────────────────────────────┤
│  ## 输出格式                     │  ← 结果长什么样
├─────────────────────────────────┤
│  ## 约束                         │  ← 不该做什么、边界在哪
└─────────────────────────────────┘
```

**逐字段解释**：
1. **文件位置**：`~/.claude/skills/`（全局）或项目内 `.claude/skills/`（项目级）
2. **文件格式**：`.md` 文件（Markdown，纯文本，用任何文本编辑器都能打开）
3. **文件名**：去掉 `.md` 后缀就是你的斜杠命令名（如 `meeting-notes.md` → `/meeting-notes`）
4. **关键组成部分**：
   - `# 标题` — Skill 的名称
   - 描述段落 — 一句话说明用途
   - `Use when...` — 触发说明，告诉 Claude 什么情况下该用这个 Skill
   - `## 工作流程` — AI 应该做什么，按什么顺序做
   - `## 输出格式` — 结果的结构模板
   - `## 约束` — 边界条件和限制

**实操**：学员打开课程提供的示例 Skill 文件，用不同颜色高亮标注每个部分。

**产出**：学员能指出 Skill 文件中每个部分的作用，并解释"Use when"的含义。

---

### Section 4：改造现成 Skill（25 min）

**目标**：通过小修改建立"我能改"的信心。

**改造任务（循序渐进）**：

**第一步：改文字（5 min）**
- 打开课程提供的"会议纪要整理" Skill 文件
- 修改输出格式（如把表格改成列表）
- 修改约束（如把"不超过原文 30%"改成"不超过 500 字"）
- 保存，用 `/meeting-notes` 测试效果

**第二步：改行为（10 min）**
- 在工作流程中增加一个步骤（如"6. 为每个待办事项评估优先级"）
- 修改输出格式，增加一个新的输出区块
- 保存，测试效果

**第三步：改触发说明（10 min）**
- 修改 `Use when...` 这一行，添加新的触发关键词
- 例如：把 `"整理会议纪要"` 改成 `"整理会议纪要", "会议记录", "meeting summary"`
- 保存，用新的关键词测试是否能触发

**教学重点**：
- 改完立刻测试，形成"改→测→看效果"的循环
- 强调：改坏了不要紧，Skill 文件随时可以恢复（课程提供原始备份）
- 每次只改一个地方，确认效果后再改下一个

**常见问题处理**：
- 改完没效果？→ 确认文件已保存，重新启动 Claude Code 会话
- 输出格式乱了？→ 检查 Markdown 格式是否正确（缩进、空行）
- 不确定改哪里？→ 对照 Section 3 的结构图定位

**产出**：学员完成至少 2 处修改并验证效果。

---

### Section 5：从零创建 Skill（50 min）

**目标**：从空白文件开始，写出一个完整可用的 Skill。

**教学方法**：提供 3 个场景模板，学员选一个跟着做。

#### 可选案例 A：周报生成 Skill

**场景**：每周五把本周工作笔记整理成结构化周报
**Skill 功能**：
- 输入：学员粘贴本周的工作笔记/日报（纯文本）
- 按"本周完成 / 下周计划 / 需要协助"三段式输出
- 自动控制篇幅在 300 字以内

**填空式模板**：
```markdown
# 周报生成

[一句话描述：把本周工作笔记整理成结构化周报]

Use when the user asks to "生成周报", "weekly report", or pastes work notes asking for summary.

## 工作流程

1. 从用户粘贴的文本中提取本周完成的工作事项
2. 识别下周计划相关的内容
3. 提取需要协助/阻塞的事项
4. 按三段式结构组织输出

## 输出格式

### 本周完成
- [要点列表，每项一句话]

### 下周计划
- [要点列表]

### 需要协助
- [如无则写"暂无"]

## 约束

- 总字数不超过 300 字
- 不添加原文中没有的工作内容
- 语言风格：[填入你的偏好，如"简洁专业"/"轻松口语"]
- 使用中文输出
```

#### 可选案例 C：内容改写 Skill

**场景**：把一篇文章改写为不同平台的风格
**Skill 功能**：
- 输入：原始文章
- 参数：目标平台（公众号 / 小红书 / LinkedIn）
- 输出：适配目标平台风格和字数的改写版本

**填空式模板**：
```markdown
# 内容改写

[一句话描述：把文章改写为指定平台的风格]

Use when the user asks to "改写", "rewrite for platform", or provides text with a target platform name.

## 工作流程

1. 分析原始文章的核心观点和关键信息
2. 根据目标平台确定风格和字数要求：
   - 公众号：深度长文，1500-3000 字，段落分明，适当使用小标题
   - 小红书：轻松口语，300-800 字，多用 emoji，开头要有 hook
   - LinkedIn：专业简洁，500-1000 字，英文为主，突出洞察和数据
3. 按目标平台风格重写内容
4. 检查字数是否在范围内

## 输出格式

### 平台：[目标平台名]
### 字数：[实际字数]

---

[改写后的正文]

---

### 改写说明
- 保留了哪些核心观点
- 做了哪些风格调整

## 约束

- 不改变原文的核心观点和事实
- 不添加原文中没有的数据或引用
- 如果用户未指定平台，询问目标平台
- [填入你的偏好：如"默认使用中文"/"保持原文语言"]
```

**统一教学步骤（无论选哪个案例）**：

```
Step 1：明确需求（5 min）
  → 这个 Skill 要解决什么问题？
  → 输入是什么？输出是什么？

Step 2：创建文件（5 min）
  → 在 .claude/skills/ 目录下创建 .md 文件
  → 文件名 = 你想要的斜杠命令名（如 weekly-report.md → /weekly-report）

Step 3：编写标题和触发说明（5 min）
  → # 标题
  → 一句话描述
  → Use when... 触发条件

Step 4：编写工作流程（10 min）
  → 用自然语言描述 AI 应该做什么
  → 分步骤写清楚流程（编号列表）

Step 5：定义输出格式（10 min）
  → 用 Markdown 写出输出的结构模板
  → 包含占位符说明每个部分填什么

Step 6：添加约束（5 min）
  → 什么不该做
  → 字数/格式/语言限制

Step 7：测试验证（10 min）
  → 用 /skill-name 调用
  → 检查输出是否符合预期
  → 至少测试 3 次，确认结果稳定
  → 根据结果微调措辞
```

**常见问题处理**：
- 文件创建困难：提供终端命令 `touch .claude/skills/my-skill.md` 和文本编辑器打开方式
- 不知道写什么：直接复制模板，只改方括号 `[...]` 里的内容
- 测试结果不理想：对照"约束"部分，看是否遗漏了关键限制

**产出**：学员写出一个完整的 Skill 文件并成功调用，连续 3 次输出质量稳定。

---

### Section 6：测试、迭代与分享（15 min）

**目标**：掌握 Skill 的调试方法和分享方式。

**调试技巧**：
1. 常见问题排查清单：
   - Skill 没被识别？→ 检查文件位置和命名
   - 输出不符合预期？→ 检查指令是否足够具体
   - AI 忽略了某些约束？→ 用更强的措辞或加 IMPORTANT 标记
2. 迭代方法：每次只调一个变量，对比前后效果

**分享方式**：
1. 文件分享：直接把 .md 文件发给同事
2. Git 仓库：放在项目的 `.claude/skills/` 目录，团队共享
3. 社区分享：发布到 GitHub 或相关社区

**进阶方向预告**（不展开，仅告知存在）：
- Skill 可以引用其他 Skill
- Skill 可以配合 MCP Server 获得更多能力
- Skill 可以设置复杂的条件逻辑

**产出**：学员掌握调试方法，知道如何分享自己的 Skill。
